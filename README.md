Задание 1.
Выяснить ограничения традиционных ноутбуков Collab при разработке 
ML-систем по сравнению с Marimo.

Классические ноутбуки, такие как Google Colab и Jupyter, ориентированы на 
линейное выполнение ячеек, тогда как Marimo использует граф зависимостей.
В ноутбуках порядок выполнения задаётся вручную, а в Marimo пересчёт 
происходит автоматически при изменении данных. Ноутбуки хранят состояние 
в памяти, что может приводить к ошибкам и несогласованности.
В Marimo состояние вычисляется заново, обеспечивая предсказуемость результатов.
С точки зрения интерфейса ноутбуки в основном статичны, а в Marimo есть 
реактивные UI-элементы, связанные с кодом. Однако такая модель требует 
строгого соблюдения зависимостей: UI-элементы нужно создавать в одной 
ячейке, а использовать в другой, иначе возникают ошибки выполнения. 
  
Еще можно добавить. Фишка SQL-ячеек, можно добавлять подключения к БД, и
работа с БД будет встроена в RAG. Формат .ipynb неудобен для Git и 
командной разработки, в отличие от .py файлов Marimo.
Ноутбуки часто используются как смесь кода и заметок, но их сложно 
поддерживать как документацию.
В Marimo код и пояснения можно использовать и как презентацию, и как 
документацию (режим Toggle App View).
Итого, к Marimo нужно привыкнуть после анархии 
обычных notebook. Здесь переменные объявляются один раз, а зависимости 
между ячейками автоматически управляют порядком выполнения, 
нельзя читать значения UI в той же ячейке и 
дублировать переменные. Все нюансы не успел опробовать, надо работать с ним. 

Задание 2. Обеспечить воспроизводимость эксперимента, используя DVC/MLflow.

Структура
prepare → train → evaluate
prepare → train.csv, test.csv
train → model.pkl
evaluate → metrics.json

⚙️ Параметры
train:
  model_type: random_forest
  random_forest:
    n_estimators: 200

Использование:

dvc exp run --temp -S train.random_forest.n_estimators=300

🧪 Эксперименты
dvc exp run --temp ...
dvc exp show --only-changed
dvc exp branch <exp>
git checkout <exp>
dvc checkout

Ключевое:

используется --temp → не ломает workspace
exp apply на Windows нестабилен → заменён на exp branch

📈 MLflow
mlflow server --backend-store-uri sqlite:///mlflow.db --port 5000

Логируется:
параметры
метрики

датасеты

Недостатки DVC и решения

Неочевидное workflow с Git (exp vs workspace) — dvc exp run загрязняет 
workspace → использован dvc exp run --temp и переход через 
dvc exp branch + git checkout

Проблемы с lock-файлами на Windows — .dvc/tmp/lock блокирует 
команды → отказ от exp apply и переход на exp branch + запуск через .bat

Слишком избыточный вывод dvc exp show — много лишних колонок → использован 
--only-changed и перевод metrics.json в metrics

Чувствительность к структуре params.yaml — ошибки из-за неверных 
путей → использованы полные пути параметров 
(train.random_forest.n_estimators) и params: - train


Driver Stats Feature Store 🚀
Этот проект реализует Feature Store для управления признаками статистики водителей с использованием Feast и PostgreSQL. Проект переведен с файлового хранилища (Parquet) на полноценную БД.
🏗 Структура проекта

    local_repo/feature_repo/ — Основной репозиторий Feast.
        feature_definitions.py — Описания сущностей (Entity) и признаков (Feature Views).
        feature_store.yaml — Конфигурация хранилищ (Offline/Online Store).
    scripts/ — Вспомогательные Python-скрипты.
        seed_postgres.py — Генерация тестовых данных и наполнение PostgreSQL.
        apply.py — Скрипт для безопасного применения конфигурации Feast.
        materialize.py — Скрипт для переливки данных из истории в Online Store.
    .env — Файл с секретами (логины, пароли, хосты). Не комитить!

⚙️ Настройка окружения

    Виртуальное окружение:
    Убедитесь, что используете локальный venv.
    Переменные окружения:
    Создайте файл .env в корне проекта:
    ini

    DB_USER=tuser
    DB_PASSWORD=12345
    DB_HOST=localhost
    DB_PORT=5432
    DB_NAME=mydb

    Use code with caution.
    Конфигурация Feast:
    В feature_store.yaml настроено использование PostgreSQL с отключенным SSL (sslmode: disable) для совместимости с локальной БД.

Все скрипты рекомендуется запускать из корня проекта через интерпретатор вашего окружения.

Задание 5

Источник данных (Client Data)
Это точка входа: видео с камер наблюдения, регистраторов или архивов.

    Как работает: Данные поступают по API или через загрузку в интерфейс.
    Альтернативы:
        Real-time Stream (RTSP): Обработка прямого эфира с камер (задержка критична).
        Batch Upload: Загрузка архивов за сутки (приоритет на объем, а не на скорость).

2. Хранилище сырых данных (S3 Raw Video)

    Как работает: Промежуточное "озеро данных" (Data Lake). Позволяет хранить исходники для аудита или повторной обработки.
    Альтернативы:
        NFS/Shared Volume: Если всё крутится на одном мощном сервере (дешевле, но плохо масштабируется).
        In-Memory (Redis): Для очень коротких роликов, чтобы не трогать диск.

3. Оркестратор / Splitter (FFmpeg)
Самый важный узел для параллелизма. Он режет видео на части (чанки) по 10-60 секунд.

    Преимущество: Если видео идет час, его можно раздать на 60 разных GPU и обработать за минуту.
    Альтернативы:
        Frame-by-Frame: Резать не на видео-куски, а на отдельные картинки (JPG/PNG).
            Минус: Огромная нагрузка на диск и потеря метаданных звука.
        Scene Detection: Резать видео по моментам смены кадра.
            Плюс: Помогает трекингу лиц (лицо не "прыгает" на стыке чанков).

4. Очередь задач (Task Queue / SQS)
Это "буфер", где лежат ссылки на чанки, ожидающие обработки.

    Как работает: Гарантирует, что если один Worker "умрет", задача не пропадет, а вернется в очередь.
    Альтернативы:
        RabbitMQ: Позволяет более гибко настраивать приоритеты (например, видео от VIP-клиентов обрабатывать быстрее).
        Kafka: Если видео идет бесконечным потоком.

5. Пул воркеров (GPU Workers)
Ядро системы, где происходит ML-магия. Вместо вашего синхронного кода здесь работает инференс модели.

    Как работает: Забирает чанк, прогоняет через детектор лиц (например, YOLOv8) и накладывает блюр.
    Альтернативы (Детекторы):
        Haar Cascades (ваш код): Очень быстро, но плохо ловит лица в профиль и часто ошибается.
        Mediapipe: Оптимальный баланс "скорость/качество", отлично работает на CPU и GPU.
        YOLOv8-face: Самая высокая точность, но требует мощных видеокарт.

6. Хранилище обработанных данных (S3 Blurred Video)
Здесь лежат заблюренные чанки.

    Преимущество: Позволяет хранить результат перед сборкой, чтобы не перегружать оперативную память Merger-а.

7. Сборщик (Merger / FFmpeg)
Соединяет чанки обратно в один файл.

    Как работает: Склеивает видео без перекодирования (-c copy) и возвращает оригинальный звук.
    Альтернативы:
        On-the-fly Assembly: Склеивать видео в момент запроса пользователем на просмотр.
            Плюс: Экономия места.
            Минус: Высокая нагрузка при каждом просмотре.
